# MAS Startup Eval

[Русский](README.md) | [English](README.en.md)

Мультиагентная система на Claude API для оценки перспективности российских технологических стартапов (0–3: закрылся / выживает без роста / растёт / лидер-exit).

## Данные

### Объём выборки

`Data/Startups.xlsx` — один файл, один лист, уже дедуплицированный:

- TOP: 168 строк
- NOTOP (контроль): 184 строки
- Итого: 352 строки

### Что значит TOP / NOTOP — важные нюансы

- **TOP** — стартапы, отобранные **экспертами** в топ программы поддержки по итогам конкретного годового цикла. Это НЕ то же самое, что топ-50 по числовому рейтингу платформы — экспертный отбор и числовой ранг могут расходиться, это нормально и не является ошибкой данных.
- **NOTOP** — стартапы, отобранные экспертами как «вне топ-50». По той же причине среди них тоже встречаются записи с высоким числовым рангом — не баг.
- **`год`** — год конкретного цикла/волны отбора (2022–2025), не год основания компании.
- Один и тот же стартап может законно участвовать в разных годовых циклах и в один год оказаться в TOP, а в другой — в NOTOP (экспертное признание меняется год к году).
- Числовой ранг платформы — поле приватных мастер-таблиц, в `Data/Startups.xlsx` его нет; здесь это дано только как контекст методологии отбора.

### Дедупликация: правила и что сделано

**Легитимные повторы (не дубли, не трогали):** один и тот же основатель может подавать разные проекты в разные годы — это разные стартапы одного серийного предпринимателя, оставлены как независимые строки (например: один и тот же человек — три разных проекта в 2023/2024/2025).

**Настоящие дубли (удалены):** дублем считалась запись, где совпадают основатель+вуз **и** суть/название проекта — один и тот же стартап, переоценённый в разные циклы с расходящимся уровнем исхода. Правило разрешения конфликта:

1. Если дубль пересекает границу TOP/NOTOP — приоритет у **TOP**-записи (топ-статус приоритетнее вне зависимости от года).
2. Если дубль внутри одной группы (TOP↔TOP или NOTOP↔NOTOP за разные годы) — приоритет у **более позднего года** (отражает актуальный статус).
3. Если различается статус верификации — приоритет у верифицированной записи.

Всего удалено 16 строк (2 — при проверке пересечений между TOP и NOTOP, 14 — при расширенном поиске по основатель+вуз).

### Поля: что физически есть, что видят агенты, что скрыто

| Колонка | Смысл | Видят агенты? |
|---|---|---|
| `Name`, `год` | Идентификатор, год цикла отбора | да |
| `суть_проекта`, `ключевые_слова_nlp` | Описание проекта (100% заполнено) | да |
| `рынок` | Текст о рыночном/регуляторном контексте — **не готовая категория рынка** (~94% заполнено) | да |
| `инвестиции` | Текст о потребности в инвестициях — **не факт привлечённых денег** (~52% заполнено) | да |
| `основатель`, `вуз`, `регион` | Метаданные команды | да |
| `pat_rus_*`, `pub_*`, `pat_foreign_*` | Патенты РФ / научные публикации / зарубежные патенты по теме (не по самому стартапу), 2018–2024 | да |
| `Уровень_исхода_0_3` | Реальный исход (0–3) | **нет** — в `Data/Startups.xlsx` физически отсутствует, хранится в отдельном приватном файле для валидации после прогона |
| `Верифицирован`, `Источник_исхода`, `Комментарий_исхода`, `url` | Метаданные разметки исхода | **нет**, только в приватных мастер-таблицах, в репозиторий не попадают |
| Группа (топ/контроль) | Принадлежность к выборке | **нет** — только в приватном файле для валидации |

## Структура

```
prompts/     системные промпты 5 агентов (Scout → Technologist/Market/Team → Integrator)
src/         пайплайн (src/pipeline.py)
Data/        Startups.xlsx — входная таблица со стартапами
results/     результаты прогона (JSON на стартап + results.csv, не в репо)
```

## Запуск

```
pip install -r requirements.txt
cp .env.example .env   # вписать OPENROUTER_API_KEY

python src/pipeline.py --input Data/Startups.xlsx --limit 20
```

Флаги:

- `--input` — путь к таблице (xlsx/csv)
- `--output-dir` — папка результатов (по умолчанию `results`)
- `--limit` — ограничить число стартапов (для отладки)
- `--model` — модель для всех агентов (по умолчанию anthropic/claude-sonnet-4.6)
- `--mode mas|single` — полная мультиагентная схема или один LLM-вызов со всеми полями (абляция мультиагентности)
- `--skip-agent scout|technologist|market|team` — пропустить агентов, можно несколько сразу (абляции); нельзя пропустить все четыре одновременно
- `--force` — пересчитать всё заново, игнорируя кэш

Воспроизводимость: параметры генерации фиксированы (temperature=0.0, top_p=1.0, seed=42, max_tokens=2048) и записываются в блок `_meta` каждого JSON вместе с sha256-хэшами промптов, датой и расходом токенов по каждому агенту (`_agent_meta`). Кэш привязан к комбинации Name+модель+режим+набор пропущенных агентов: прогоны разных конфигураций в один `--output-dir` не смешиваются, но лучше давать каждой конфигурации свою папку.

## Архитектура

Агенты не выходят в интернет и не видят реальный исход или принадлежность к группе — только поля датасета, явно перечисленные в белых списках в `pipeline.py`. Каждый агент возвращает JSON по фиксированной схеме (`output_config.format`), поэтому парсинг ответа надёжен без ретраев на кривой JSON.

| Агент | Получает из таблицы | + |
|---|---|---|
| Scout | Name, суть_проекта, ключевые_слова_nlp, рынок, основатель, вуз, регион, год | — |
| Technologist | pat_rus_\*, pub_\*, pat_foreign_\* | профиль Scout |
| Market | рынок, инвестиции, регион, год | профиль Scout |
| Team | основатель, вуз, регион, год | профиль Scout |
| Integrator | — | профиль Scout + выводы трёх специалистов |

## Валидация

`Уровень_исхода_0_3` (реальный исход) и метка группы (топ/контроль) хранятся в отдельном приватном файле, не в `Data/`. После прогона сверяются с оценкой Интегратора:

- F1, recall, quadratic weighted kappa (QWK — предпочтительна для порядковой шкалы 0–3)
- Сравнение оценок между топ- и контрольной группой (ROC/AUC, тест Манна-Уитни) — ключевая гипотеза исследования
- Опционально: число попаданий одного основателя в топ-рейтинги за разные годы — как дополнительный признак серийного предпринимательства

## Известные ограничения / нюансы качества данных

- В NOTOP **все** записи с `Уровень_исхода=0` одновременно не верифицированы (100% совпадение). Уровень 0 в контрольной выборке функционально означает «независимых данных не найдено», а не подтверждённое отсутствие роста/закрытие — риск для калибровки Интегратора: модель может выучить «мало упоминаний в сети» вместо «признаков провала».
- В патентных полях (`pat_rus_*`, `pat_foreign_*`, `pub_*`) нет ни одного пропуска — только легитимные нулевые значения (патентов по ключевым словам не найдено).
- Сводный численный TTS-показатель (взвешенная сумма z-нормализованных объёмов и темпов роста патентов/публикаций) рассчитывался отдельным офлайн-скриптом вне пайплайна и в экспериментах не показал предсказательной силы ни для реального исхода, ни для попадания в топ (отрицательный результат, обсуждается в статье). В пайплайне Технологу передаются категориальные индикаторы трендов из `preprocessing.compute_trend()` (total, growth_ratio, low_base, trend_label), а не численный TTS.
